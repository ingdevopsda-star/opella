# Azure Infrastructure with Terraform

Reusable, secure, and maintainable IaC for provisioning Azure infrastructure across multiple environments using Terraform and GitHub Actions.

---

## Repository Structure

```
bootstrap/              → one-time remote state backend (run before any environment)
modules/
  vnet/                 → reusable VNET module
environments/
  dev/                  → dev: eastus, Standard_B1s, LRS storage
  prod/                 → prod: westeurope, Standard_B2s, GRS storage
.github/workflows/
  terraform-ci.yml      → plan on every pull request
  terraform-cd.yml      → apply on merge to main, manual approval for prod
```

## Reusable VNET Module

The module provisions a Virtual Network with configurable subnets, one NSG per subnet, and optional service endpoints. The caller defines the full topology — the module encodes no environment-specific assumptions.

```hcl
module "vnet" {
  source              = "../../modules/vnet"
  name                = "vnet-dev-eastus"
  location            = "eastus"
  resource_group_name = azurerm_resource_group.this.name
  address_space       = ["10.0.0.0/16"]

  subnets = {
    "snet-vm" = {
      cidr_block        = "10.0.1.0/24"
      service_endpoints = []
      nsg_rules = {
        "allow-ssh" = {
          priority                   = 100
          direction                  = "Inbound"
          access                     = "Allow"
          protocol                   = "Tcp"
          source_port_range          = "*"
          destination_port_range     = "22"
          source_address_prefix      = "1.2.3.4/32"
          destination_address_prefix = "*"
        }
      }
    }
    "snet-storage" = {
      cidr_block        = "10.0.2.0/24"
      service_endpoints = ["Microsoft.Storage"]
      nsg_rules         = {}
    }
  }

  tags = { environment = "dev", managed_by = "terraform" }
}
```

Full variable and output reference: [`modules/vnet/README.md`](modules/vnet/README.md) — auto-generated by `terraform-docs`.

## Environments

Both environments use the same module call. Only the values differ.

| | dev | prod |
|---|---|---|
| Region | eastus | westeurope |
| VM size | Standard_B1s | Standard_B2s |
| Storage replication | LRS | GRS |
| OS disk | Standard_LRS | Premium_LRS |
| Public IP | Yes | No |
| SSH access | Restricted CIDR | Deny all inbound |

### Resources per environment

- **Resource Group** — RBAC boundary, tagging scope
- **VNET** (via module) — subnets: `snet-vm`, `snet-storage`
- **Linux VM** — Ubuntu 22.04 LTS, SSH key authentication only
- **Storage Account + Blob container** — private, service endpoint from `snet-storage`, public blob access disabled

## Design Decisions

### Resource Groups vs Subscriptions

**Resource Groups** are used — one per environment. They provide sufficient RBAC isolation and billing visibility for this scope while remaining compatible with a single free-tier subscription.

At organizational scale, **separate subscriptions** per environment are the right choice: hard quota isolation, independent audit logs, and compliance boundaries enforced via Azure Management Groups (Cloud Adoption Framework).

### Directory-per-environment vs Workspaces

**Directory-per-environment** is used instead of Terraform workspaces. Workspaces share the same `.tf` files and require workspace-based conditionals as environments diverge — fragile at scale. Separate directories give each environment its own state, its own variables, and its own blast radius.

### Tagging

All resources merge a `local.common_tags` block (environment, region, managed_by, project). Tags are enforced structurally — there is no code path that creates a resource without them.

### Authentication

GitHub Actions authenticates to Azure via **OIDC federated identity** — no `ARM_CLIENT_SECRET` stored anywhere. The pipeline exchanges a GitHub-signed JWT for an Azure token via a federated credential on the App Registration.

## CI/CD Pipeline

```
Feature branch  →  PR opened
                   └── fmt + validate + tflint + plan
                       plan output posted as PR comment

PR merged to main
                   └── terraform apply dev (automatic)

workflow_dispatch (deploy_prod=true)
                   └── terraform apply dev → [manual approval] → terraform apply prod
```

Production deploys require a human approval via **GitHub Environment protection rules** (`Required reviewers` on the `prod` environment).

Required GitHub Secrets:

| Secret | Description |
|---|---|
| `AZURE_CLIENT_ID` | App Registration client ID |
| `AZURE_TENANT_ID` | Azure AD tenant ID |
| `AZURE_SUBSCRIPTION_ID` | Target subscription ID |
| `VM_SSH_PUBLIC_KEY` | SSH public key for VM access |

## Code Quality

Checks run locally via `pre-commit` on every commit, and in CI on every pull request.

| Tool | Role |
|---|---|
| `terraform fmt` | Canonical formatting |
| `terraform validate` | Syntax and type checking |
| `tflint` + azurerm plugin | Best practices, deprecated resources |
| `terraform-docs` | Auto-generates `modules/vnet/README.md` |
| `detect-private-key` | Blocks accidental secret commits |

```bash
pip install pre-commit && pre-commit install
```
